第一份工作总结

产品 个人总结
产品经理
发布日期:   2020-10-02

由于自己的操作失误，之前博客的本地配置文件被删除了。（其实是被GitHub库反向同步了）不过幸好图片文件都是上传cdn的，所以趁着国庆有空就重新配置了一下。（配置完发现图片虽然还存在服务器里，网页上却显示不出来orz）这次在GitHub多设了一个备份仓库，希望人没事。

第一份工作是七猫的助理产品经理。回头看来，自己这几个月没有了解多少 SDK 相关的工作，也没有负责数据周报等全面的统计和分析，竞品分析也寥寥，这么看来确实还有很多不足。不过还是要总结一下自己这几个月的工作内容与得失的。

Part1：产品基本功

1.用户反馈

张小龙做 QQ 邮箱时候的“十百千”，每个月深度访谈十个用户，阅读一百篇相关领域的文献，在平台上就相关问题互动一千次。单纯在后台回复用户的反馈自然是达不到这个数量的，不过这也是最基础的部分了。

处理用户反馈，首先掌握了客服回复的话术；其次，了解了应用目前需要改进的方面；同事、朋友的反馈也属于一种用户反馈，综合这些反馈就形成了个人的需求池。阅读器的一些功能，比如自动翻页、上下翻页、横屏阅读、账号同步等，大家都知道这些还没有做或者做得不够好，也在策划相应的功能。（对应功能什么时候上线就是另一回事了。）

我当时比较喜欢处理的一类反馈是用户拿广告推广的截图来问是什么小说。喜欢的其实是通过主角名、章节内容等边边角角的信息去搜索小说、还有在百度搜索和书库里核对不同书名作者名的感觉，偶尔还会发现基本抄袭的书。不过短期来看这样的反馈对于构建需求池来说是很低效的，用户想找的小说五花八门，而解决方法其实是增加一个设备识别让通过推广素材进入应用的用户可以直接跳转对应书籍，这和在反馈里直接告诉用户书名并无关联。还有一类反馈是找不到想要的书——这其实要优化应用的搜索机制，在用户精确查找没有结果的时候，推荐同类型、同情节等用户可能喜欢的书。这些方面都是做得还不够的。

还有在敏捷的思路下，把用户反馈转变成用户故事。把用户的问题精炼成一个场景的问题，通过用户故事的形式表达出来，让产品、开发、测试等所有参与需求的成员都能够理解需求的目的、背景，以及在自己负责的方面应该如何去做。

2.竞品分析

主管是这么和我说的，设计一个功能不要靠自己去想，去比较一下市面上的竞品，总结出一个合适的方案就行。只能说我参与策划的功能不多，主要是后台方面，也没有做很多相关的比较吧。

我主要做了两个竞品分析，一个是实习期的作业，比较了七猫、番茄和 QQ 阅读三个阅读软件的异同；另一个是排行榜推荐机制的比较。第一个主要从阅读器方面入手，每家阅读软件的阅读功能都有区别，QQ 阅读比较全面，有章评、句评，还有很多互动功能，以及会员、集卡等偏向社交的功能。番茄和七猫性质、类型、模式都比较相似，阅读器最大的区别应该是风格。从番茄的阅读器上是可以看出设计感的（虽然最有设计感的应该是微信阅读，听书最好的也是微信阅读），阅读比较赏心悦目。番茄编辑部还专门写了一本番茄手册，每周更新，官方和用户的互动很良性。七猫就更像一个直男，还有一些土，但是也很直接。七猫的一个特色是单独做了一个“菜单”按钮，防止用户不知道点击屏幕中间就可以呼出菜单的情景下找不到菜单的问题。《向往的生活》里有一期的推广里，嘉宾就是点那个菜单按钮呼出菜单的。另外，具体比较起来，七猫的广告频率和打扰程度都没有番茄那么高。还有一个小细节，从后往前翻页的话，七猫只要前一页不是广告，就不会再翻到广告，番茄会有（偷笑）。目前距离比较竞品也过去几个月了，两家都在广告上做了很多改变和尝试，这样的比较时效性还是比较短的。还想多说一句，轻之文库的排版和翻页非常流畅，支持章内所有页面的预览，这对开发的技术要求是很高的。

第二个竞品分析关于推荐机制，这就不限于阅读软件了，所有包含推荐列表的应用、网站都是可以参考的对象。我印象比较深的是动漫之家的编辑推荐，经常会蹭一些热点，找一些新奇搞笑的梗。还有就是B站的推荐计算公式，三连、播放、分享、弹幕评论分别乘不同的权重相加。后来我设计的评分公式也是根据这样的思路来计算的。

3.会议纪要

会议纪要大概不能算产品的基本技能，而是对新人的训练。邮件奇妙的格式折磨了我很久，有的字体在文本编辑的时候显得很大，发送出去却变成了小字体。正文编辑，我已经从 word 转 markdown 了，稍微方便了一点点。如果没有开头的表格，我还可以省下每次至少半小时的修改格式的时间。有同事推荐 outlook，不过我还在尝试中。

小结

单单说产品的基本功是处理用户反馈和进行竞品分析好像不太贴切。应该总结成“发现需求”和“策划需求”两方面。用户反馈是发现需求的一种方法，竞品分析是策划需求的一种手段。此外，就是我觉得目前的自己有所欠缺的创造权，或者说，在设计中加上自己的审美和思考。不能说我没有审美和思考吧，但是好像策划和设计的东西没有自己的风格，就让人觉得，任何一个人都可以做到这种程度。应该好好思考如何成为一个设计者，而不是软件操作员。

Part2：书城功能

1.埋点迁移核对

这部分是比较枯燥的工作，核对书城埋点是否全面，以及埋点数据是否准确。不过也是通过这个工作，以及跑数据验证排行榜规则，熟悉了 excel 的操作。同时，核对埋点也是对应用的一种剖析，每个页面、每个按钮都要了解，比刚入职做的应用逻辑梳理会更全面一些。

2.排行榜设计

排行榜设计最早是从一个数值显示开始的。有一个排行榜数据展示不合理，最早以为是该排行榜采用了单一的阅读数据排序，而该阅读数据逼近极限值，导致排行榜顶部书籍展示的数据过于接近。后来发现前面数据不正常的书籍都是运营配书，数据问题是配书的数据计算公式的问题。我去了解了一下这部分公式的计算逻辑，并且根据这个逻辑，设计了最小限度的修改。

之后排行榜的问题就是最根本的问题——如何在排行榜上展示读者想看到的书。也是和运营讨论，设计了一些排行榜书籍的剔除规则，把质量不高的书从排行榜剔除出去。不过这个规则并不是很完善，偶尔还会有比较差的书上榜，所以根据不同排行榜的情况进行了单独的调整。再之后，就是在后台设计了一个排行榜屏蔽的页面，可以在后台看到排行榜上展示的书籍的数据，并且把质量低的书一键加入黑名单。排行榜这部分目前还是需要人工管控，不能完全依靠系统取书。

第三点是重新设计书籍评级的公式。因为现在书库除了网络小说，还引进了很多出版读物，之前的公式套用在出版读物上得分都不高，这样的评价并不准确。实际上新设立一个公式并没有解决这个问题（笑）。但是确实需要一个新的公式，重新订立评级，也是针对部分排行榜取书数据单一的问题，增加一个参考标准。

最后是新的排行榜设计。字数较少的连载书、平台原创的书籍，需要增加曝光，所以单独设计一个榜单。增加了图书板块、增加了评论功能，所以也增设相关的榜单。这部分我一开始的理解比较简单，按照之前排行榜的样式，修改对应榜单的取书逻辑和展示规则。但是没有考虑到很多实际情况，比如评论是新增功能，目前还有很多不确定因素，而且目前评论总量偏低，数据可能不是很好看；以及图书方面没有好的评价标准，目前不一定能筛选出合适的书。针对这些问题，我也是重新策划了对应需求，进行了两次评审，之后还会根据评论、评级等功能的变化再次调整。

3.分类页、标签页注释

分类和标签的区分其实也是一个难题，不过给所有的分类和标签加上注释则是一个相对简单的工作。这个注释会在应用内的对应页面展示，切换标签/分类也会切换对应的描述。和负责的其它功能的开发不同的是多了一个 UI 设计的环节，对比竞品（主要是番茄）后选择了一个颜色偏淡的样式。由于这个需求比较简单也很好理解，开发、测试都很快。对应的，我也设计了管理注释的后台页面，管理每个分类、标签的ID、名称以及描述内容。

该功能上线后还发生了一个小 BUG。分类板块会推荐一些分类和标签，点击可以跳转对应的页面。但是有一次，跳转页面的标题和注释、页面内容都不符。当时我马上反馈了测试，联系服务端和客户端排查 BUG，但是都没有找到原因。当天晚些时候发现是运营在配置推荐标签的时候，把标题和实际推荐内容弄错了。服务端和客户端是没有错的，管理注释的后台也是没有错的，终究还是运营的操作不够仔细。不过产品策划之初就应该多考虑一些防止误操作的设计，应该好好衡量系统和人工之间的度。

4.敏捷迭代

在职的那段时间刚好是从瀑布转敏捷的尝试期吧。一个很直观的变化就是晨会从整个产品部门一起开变成了负责书城内容的产品、开发、测试一起开。敏捷会更聚焦于工作进度吧，每天要完成什么事情都更加清晰。之前提到的用户故事也是敏捷迭代的一环。敏捷迭代的另一个要求就是要把所有需求都拆分成小需求。每次迭代发布相隔两周或者一周，除去测试时间，除去开发评审、自测的时间，实际开发的时间并不长，所以需求要尽可能小，才能尽快完成。也许以后会变成每天发布呢。

小结

书城这一块工作内容我其实挺喜欢的。埋点数据虽然枯燥，但是是必不可少的一部分，我从中学到了很多。书城推荐就是真正在为用户思考，如何才能让读者轻松找到他想看的书，相关的用户反馈其实也是五花八门。如果我可以设计出新的排行榜，那一定很有意思。

Part3：审核后台

1.主流程优化

审核后台其实才是我接手的第一个比较大的需求。主流程优化，是将审核流程从“初审-复审”修改为“审核-对接-质检”，对接专门负责将内容驳回给内容方并跟进修改，把没有问题的内容提交质检。这个流程是审核讨论的结果，我设计了一个对接人的页面，遵循的还是尽量不做大的修改的原则吧，新页面和其它页面长得比较像，然后精简了信息，把对接人不需要知道的信息舍去，还能稍微加快一点页面加载时间。当然，虽说只是加了一个页面，审核流程中其它人员的操作也有一些改变，同时，其它页面也要增加对接的信息，所以主流程设计的页面基本都有一些小改动。

还有一件小事，设计对接页面的时候，因为对接需要从书籍列表跳转到驳回库处理对应书籍，我设计了“点击列表里的处理状态可以直接跳转驳回库”的功能，却因为驳回库没有搜索功能，导致这个设计只完成了一半，虽然跳转驳回库的时候携带了书籍 ID，但是没有进行搜索。后来的某天，同事偶然发现驳回库是有搜索功能的，只是驳回库和书籍列表的对应参数不一样，前端几分钟就改好了（笑）。（吐槽：为什么当初没有发现啊！）

前两天在研究驳回子流程的时候，开发告诉我审核流程又要修改，建议先确定审核主流程再调整子流程。我其实不是很理解吧，作为产品却不知道这个消息，以及，主流程再次修改是说明之前的流程并没有提高效率吗？

2.后台 UI 优化

UI 优化，是和主流程优化几乎同时在跟进的。这部分，UI 已经给过一份样例，不过和审核多次沟通后，还是调整了很多细节（每个页面的按钮、标记配色、表格行高字号等等）。水磨工夫了大概一个月吧，做出的第一版却不尽如人意。有一部分是因为前端说有实现难度所以更换了样式但没有及时和审核沟通，有一部分是因为屏幕大小不同无法完全适配，还有一部分就是真的没有做好但是验收的时候没有发现了。第一版出来以后，除开验收没有发现的地方，其实我也感觉没有期待中那么美观的，在“打回去回炉重造还是凑合着先上线”的纠结中，最终选择了上线，然后被骂得很惨。沉重反思，以后功能没做好坚决不上线。

于是，审核后台 UI 优化一期上线后，立刻开始了二期的优化。这时候我发现，之前没有做到的地方，很多是某页面调个颜色、某页面调个字号字体这样的，基本都是调整个别参数的问题，不过由于涉及到后台整体的样式，需要找前端的老大确认。再次和 UI 核对就发现，字体颜色问题主要是一级、二级重要的文字用了不一样的黑，而前端以为的一级文字和审核期待的一级文字并不一样。（ UI 说：前端写的颜色和 UI 稿上的颜色不一样啊！）这次调整的心得就是沟通真的很重要，自以为足够的沟通，可能还远远不够。为此我也真正去做了一个礼拜的审核工作。（我的审核效率还没有专职审核人员的十分之一orz）

第三次，则是更加零碎的修改。这时候我发现之前设计的对接人页面，底部的一些按钮直接套用了以前审核页面的按钮，实际上由于对接人工作不同，很少会用到；而对接人可能会常用的按钮并没有添加上去，针对这个问题进行了按钮的调整。还有一个四级页面的标题、筛选字段、按钮的排布非常拥挤，这个问题也有些缘由：之前有一个二级页面也有布局很奇怪的问题，大屏所有组件排在同一行，小屏的话筛选字段可能会叠成四五行，非常奇怪。这个问题在二期的优化里已经调整了，解决方法是把筛选字段列在单独一行，这样即使小屏会换行，也只占用两行空间，不会有五行那么恐怖。但是这次是一个四级页面，藏得太深，也是我的疏忽，之前甚至没有发现这个页面的异常。还有，审核后台有的页面是选中筛选词后直接筛选，有的页面选中筛选词后还要点击“查询”才会筛选，这个操作没有统一，也有些奇怪。总之是一个不断发现问题、不断调整的过程。

最近我也在想，如果后台的调整是敏捷迭代，审核后台涉及的页面如此之多，应该会拆分成很多个小需求，每次解决一个小问题，也许会好一些？

3.人员分组管理

这也是一个讨论——设计——打回重做——搁置——重新提上日程的过程。最早是审核工作量统计的需求，审核统计的现状是人员排序固定，只要有审核角色，不管是否真正参与了审核工作都会被统计上。基于审核人员目前分成了几个大组的情况，我的策划是创建新的统计页面，按分组统计和展示每周、每月审核人员的工作量，因此还需要在人员管理中添加分组管理的功能。

对于统计页面的原型，我一开始想得比较简单，在复用审核进度页面的基础上增加分组展示就可以，结果被开发吐槽需求看不懂。那时候刚好同事有抓取、报错等需求比较紧急，所以我一边完善需求，一边延后了评审计划。后来审核那边又吐槽说我的原型太像草稿了，反馈给主管之后，经过多方考量，审核统计的需求就搁置了下来。至于说我的原型像草稿…我当时原型还是参考 UI 调整前的页面（后台其它页面的老原型），确实不是很好看；再者统计数据的那张表格线上展示的效果就很杂乱，我自认为原型里添加分组后，已经在一定程度上优化了观感…总之也是一个教训，自那之后我画原型基本上是照着网页精确到每一个像素来画的，保证和线上版本尽可能一样。

我觉得可以把审核统计和人员分组拆分开，先把人员分组的需求过一下评审，这个做好的话对审核后台所有人员展示的页面都有帮助。不过开发说产品这边另一位同事会出一个统一的人员管理组件，应用到所有系统后台，所以人员分组的需求也搁置了。大约三周后，开发又说，审核后台这边对于人员分组有很多定制化的需求，所以要单独做人员管理的页面。（为什么我是从开发那边接收结论，而不是从产品这边总结得出结论？）所以人员管理的需求还是从之前的需求里拆分出来，和审核后台三期的修改一起去做了。

这次其实我也重新设计了分组管理的样式。向负责人员管理通用组件的同事请教后，把人员管理和分组管理从一个页面拆成了两个页面，又在评审的讨论上修改设计成了人员管理为主页面、分组管理为子页面的形式。这次的设计应该是较为全面的，反而让审核觉得有些地方多此一举吧。比如我给分组设计了一个“是否统计数据”的权限——这是考虑到可能审核后台会给产品人员、处理举报人员也分一个组，但是他们没有实际参与审核工作；或者像我这样去体验审核流程、实际上不能算进审核里的情况——来区分统计真正的审核人员工作和闲杂人等的操作。审核组长觉得，目前的审核统计页面已经进行了筛选，只能看到初审/复审的工作量，不存在需要区分的问题。我考虑了一会，打算删掉这个，等以后需要再加上吧，以后或许还会有更好的处理方式。

小结

后台的用户就是我的同事，可以得到很直接的反馈，同时也意味着更容易了解哪些地方还有不足。我觉得很重要的一点是不要被运营或者开发牵着鼻子走。我可以是很好说话的，但是我的好说话是为了更好地理解运营和开发，从而去协调一个最优解。后台设计大部分是秉着“修补”的思路，尽可能少地改动，初期也尽可能宽松地给予权限，是一个由松到紧、由粗略到精确的过程。

Part4：总结

首先，我得到了一份工作经验，不管是简历上还是实际经历上。我了解了产品经理到底是做什么的、应该怎么做。我学会了axure、excel、TAPD 等等工具，同时也养成了工作每天写日报的习惯。虽说很多是流水账，坚持下来至少手没有生疏，会尝试去总结。我也在锻炼自己的产品思维，努力从用户的角度思考问题。虽然入职的时候就强调自己要换位思考，实际上还是思考得不够啊。

要说失去了什么的话，有几个需求还在策划中、待评审或者待开发，没法继续跟进有一些遗憾。最后一个月基本没看动画，最后几天加了很多班，最后主管还是说不合适，我的自信心确实受到了一些打击，也有一种荒废了自己的兴趣的感觉。还有就是失去了一些沟通的机会吧，公司里有几位非常强势的人，我很少和他们沟通，本来想着等自己积累的经验更丰富一些再找他们讨论、接手相应的需求，现在没有机会了。

我觉得自己所欠缺的，一个是主动性，一个是创造权。主动性是指学习不够主动，没有那么积极地去沟通自己负责的需求以外的事情，没有去深挖、去拓展很多东西。可是我又觉得，我负责跟进的需求其实不少，同时也读了一些文档，是我的效率不够高导致没有更多时间去深挖么。创造权低可能和我展现出的控制欲不强有关，而且我又希望尽可能小地改动，影响也偏小。又因为是新人，很多需求要三番五次地修改，导致了同事之间对我一定程度的不信任。有一种等别人布置需求的感觉，而不是我从自己的需求池里挑出重要的需求去做。这么看来还是要多表达，先让雷声大起来，雨点是大是小之后再考虑。

最后聊聊公司文化吧。公司文化其实是一个我入职以前完全没有考虑过的事情。公司组织在一起，靠人、靠制度，还要靠企业文化。七猫的企业文化其实是很朴素的一些道理，有一点像小学、初中时候的校训，对于这些道理我其实挺认同的。相信人性本善、从用户出发等等，其实还挺温情的。想拎出来聊的一点是总裁要求不能迷信数据，比较直觉派和接地气吧。有的公司确实是数据驱动的感觉，依靠大数据做判断。也有人认为不依靠数据的公司做不大。数据在我心中一直挺重要的吧，不过好像没有在这份工作中体现出来，讲需求的时候非常直觉派…

从一个公司离开，待遇方面没有问题，可能是对公司或者行业的认同感还不够吧。继续加油吧。
